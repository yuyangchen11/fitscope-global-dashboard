# FitScope Commerce OS 全量审计

日期：2026-06-30

## 结论

当前版本已经具备可演示的完整主链路：

`登录 -> 角色工作区 -> 市场/品类 -> SKU/订单 -> 评论证据 -> 关注对象`

数据库登录、会话、角色识别、关注持久化、双语切换、日期控件和钻取抽屉均已实现。当前还不能称为生产系统，主要原因不是缺少更多图表，而是数据口径、权限、移动端、性能和流程闭环仍不完整。

## P1：优先修复

### 1. 手机端登录被阻断

- 位置：`index.html:57-65`、`index.html:3091`。
- 现状：登录页使用 `100vh` 和 `overflow: hidden`；窄屏改成上下布局后，提交按钮落在首屏外且页面不能滚动。
- 影响：390 x 844 手机视口无法完成登录。
- 修复：移动端允许纵向滚动，登录卡片使用内容高度；侧栏改为抽屉或底部导航。

### 2. 日期筛选没有统一指标口径

- 位置：`index.html:5223-5234`。
- 现状：日期会影响订单、SKU 聚合和趋势图，但经营健康、收入、退货率等顶部 KPI 保持不变。
- 影响：同一页面同时展示不同时间窗口的数据，用户会做出错误比较。
- 修复：给每个指标声明 `time_scope`；可重算指标统一跟随全局日期，不可重算的结构指标改为模块内固定周期并明确标注。

### 3. 真实值与模型填充值混在同一汇总中

- 位置：`index.html:4592-4628`。
- 现状：可映射品类使用 Kaggle boutique 指标，未映射品类通过公式生成并标记为 `model_fill`，但界面汇总时未区分。
- 影响：管理者无法判断 GMV、转化、退货和毛利中哪些是观测值。
- 修复：指标增加 `source_type`、`dataset`、`as_of`、`coverage`、`confidence`；模型值不得与观测值无提示汇总。

### 4. 角色差异仍停留在界面层

- 现状：管理者和运营者使用相同五个页面，主要差异是默认页、排序、顶部 KPI 和少量文字。
- 影响：没有真正的页面、操作和数据范围权限；运营者仍可接触管理视角内容。
- 修复：建立后端权限矩阵，至少区分页面权限、指标权限、操作权限和行级数据范围。

### 5. 当前渲染方式无法支撑真实数据量

- 位置：`index.html:5264`。
- 现状：多次筛选和切换会调用 `renderAll()`，隐藏页面也被重建；Operations 当前同时包含约 815 个 SKU、1000 条订单及大量控件。
- 影响：数据量增加后会出现卡顿、内存占用和输入延迟。
- 修复：仅渲染活动页面；表格分页或虚拟滚动；缓存聚合结果；组件级局部更新。

### 6. 业务数据仍是浏览器可下载的静态 CSV

- 现状：公开数据适合演示，但未来公司订单、利润、客户和退货数据不能继续由浏览器直接读取。
- 影响：无法做行级权限、脱敏、审计和安全聚合。
- 修复：公开样例可保留静态文件；私有经营数据必须经过认证 API，由后端按角色和范围返回。

### 7. 关键交互不满足键盘与辅助技术要求

- 位置：`index.html:6149`、`#sku-search`、`#order-search`。
- 现状：甜甜圈路径虽有 `tabindex`，但没有可访问名称和键盘触发；可点击表格行不是按钮；抽屉、通知、账号和 AI 面板没有 dialog 语义、焦点锁定和焦点返回。
- 影响：键盘用户无法可靠完成钻取和关闭操作。
- 修复：使用原生按钮；补充 `aria-label` 与 Enter/Space 行为；弹层采用 `<dialog>` 或完整的 `role=dialog` 焦点管理。

### 8. 数据源不能支持“中国跨境服装市场”结论

- 当前来源：H&M 历史交易、UCSD 合身反馈、美国女装评论、Kaggle boutique/sales 数据。
- 适用范围：验证分析方法、交互流程和数据模型。
- 不适用范围：证明中国渠道、消费者、物流、价格和退货行为。
- 修复：产品内明确“公开代理样本”；正式研究需补充中国电商渠道、商品、订单、退款、物流和中文评论数据及许可说明。

## P2：流程与可信度

### 通知与关注

- 已完成：关注品类、商品组和 SKU，并按用户写入 SQLite。
- 缺失：已读状态刷新后恢复为未读，`readNotifications` 仅保存在内存。
- 缺失：通知由客户端当前数据即时拼装，不是真正的定时异常检测。
- 下一步：保存已读状态；建立订阅规则、基线、阈值、检测任务和对象级链接。

### 账号与安全

- 已完成：scrypt 密码哈希、HttpOnly + SameSite Strict 会话 cookie。
- 缺失：显示名称修改没有写回用户表，重新登录会恢复。
- 缺失：修改密码、忘记密码、账号管理、登录审计。
- 风险：演示账号和密码仍硬编码在前后端；本地 HTTP 不能作为正式部署。

### 订单真实性

- 当前订单来自公开样本，只含有限渠道字段，界面中的渠道是 `H&M channel 1/2` 等代理标签。
- 不应伪装成淘宝、京东或真实门店订单状态。
- 正式订单模型至少应包含订单、订单行、商品、店铺、渠道、支付、履约、退款和客户脱敏标识，并保留源系统主键。

### 语言本地化

- 英文界面壳层基本完整。
- 中文模式仍出现原始数据值：`new`、`unknown`、`straight & narrow`、`other`、`date`、原始痛点 ID 及英文评论。
- 需要建立“界面文案翻译”和“数据值词典”两层；评论原文可保留，但应提供中文摘要或按需翻译。

### AI 助手

- 位置：`index.html:4487`。
- 当前是固定关键词模板，不是模型服务，也没有检索、引用和权限隔离。
- 建议名称继续使用 **织见 / ZHI**。
- 接入模型前应标记为规则摘要；接入后每条回答必须带周期、筛选条件、来源和可追溯行。

## P3：体验与视觉

1. KPI 页面标题在 1280px 宽度下会被右侧控件挤成两行，应重新分配标题栏栅格。
2. Operations 和 Evidence 在桌面仍有约 31-51px 的全局横向溢出；横向滚动应只发生在表格容器内。
3. Overview 的合身环形图数值存在文本容器溢出。
4. 手机端不应展示完整桌面侧栏；应使用抽屉、顶部菜单或底部导航。
5. 删除没有业务含义的装饰性光晕和重复卡片，优先用表格、分栏和紧凑指标行。
6. 统一搜索框标签、按钮最小 44 x 44 点击区域和空状态文案。

## 架构风险

- `index.html` 超过 7,000 行，界面、状态、数据转换和事件逻辑集中在单文件中。
- 所有 CSV 通过一个 `Promise.all` 加载；任一文件失败会导致整体初始化失败。
- 需要拆分为数据访问层、领域计算层、状态层和页面组件层，并对每个数据源独立加载、校验和降级。

## 推荐实施顺序

1. 修复手机登录、桌面横向溢出和标题换行。
2. 统一日期口径，并显示指标血缘、刷新时间和模型填充状态。
3. 仅渲染活动页面，给 SKU、订单、证据表做分页或虚拟滚动。
4. 后端落实管理/运营权限矩阵和私有数据 API。
5. 完成键盘操作、dialog 焦点管理和图表可访问名称。
6. 建立真实关注告警和行动跟踪闭环。
7. 完成中文数据词典与评论摘要。
8. 最后接入有引用、有权限控制的织见服务并进行生产部署加固。

## 上线门槛

在宣称“可投入生产”前，至少需要通过：

- 手机和桌面主流程自动化测试。
- 指标定义、时间范围和血缘一致性测试。
- 角色越权与行级权限测试。
- CSV/API schema 校验、部分数据源失败和过期数据测试。
- 键盘导航、焦点管理和 WCAG 对比度检查。
- 大数据量性能测试和敏感数据安全审查。
